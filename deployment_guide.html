Key Concepts — Before You Start

Don't worry if these terms are new. Here's a quick plain-English explanation of every concept used across all three projects.

RAG — Retrieval-Augmented Generation

Instead of asking an AI to answer from memory, you first search your own documents, then hand the relevant text to the AI so it answers using your actual data.

Vector / Embedding

A way to turn text into a list of numbers that captures meaning. Similar sentences have similar numbers, so you can search by meaning, not just keywords.

FAISS

A fast open-source library (by Meta) that stores embeddings and lets you find the most similar ones instantly. Think of it as a "search engine for meaning".

AWS Lambda

A serverless compute service. You upload your code and AWS runs it on demand — you pay only when it executes, with no servers to manage.

API Gateway

The front door to your Lambda. It gives you a public HTTPS URL (POST /query) that clients can call — like a waiter taking orders in a restaurant.

AWS SAM

Serverless Application Model. A tool that lets you define your Lambda + API Gateway in a single YAML file (template.yaml) and deploy everything with one command.

S3 Bucket

Amazon's object storage — like Dropbox for AWS. Used here to store your documents and your pre-built FAISS index so Lambda can load them.

Warm / Cold Start

The first time Lambda runs your code it loads everything (cold start, slower). After that the container stays alive for a while — subsequent calls skip loading (warm start).

LangGraph StateGraph

A library that lets you define an AI workflow as an explicit graph of nodes (steps) and edges (transitions). Makes the flow visible, typed, and easy to extend.

Bedrock Knowledge Base

Amazon's fully managed RAG service. You upload documents to S3, Bedrock handles chunking, embedding, indexing, and retrieval automatically — zero custom code.

✅
Prerequisites for all projects: Python 3.12+, AWS CLI (aws configure), and AWS SAM CLI installed. Install SAM with brew install aws-sam-cli on Mac or from the official docs.

Agentic RAG

OpenAI · FAISS · SAM

The agent uses OpenAI function calling to decide how many times to search the knowledge base before giving an answer. It can search up to 3 times with different queries to gather enough context.

🎯
Best for: Full control over every part of the pipeline — chunking, retrieval, the search strategy, and the LLM. Good when you need OpenAI models or want to build your own retrieval logic.

How the agentic loop works: The LLM receives your question and decides to call search_knowledge_base("your query"). The result (top-5 chunks) is sent back to the LLM. The LLM can call the tool again with a different query if needed, up to 3 times. Then it writes the final answer.

Install prerequisites

Make sure Python 3.12, AWS CLI, and SAM CLI are installed. Create an S3 bucket to store the index and documents.

terminal
# Configure AWS credentials (one-time)
aws configure

# Create an S3 bucket (choose a unique name)
aws s3 mb s3://my-rag-bucket --region us-east-1
1

Local setup — install dependencies

Create a Python virtual environment and install all required packages.

Agentic_RAG/
cd Agentic_RAG
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt

Then copy the environment template and fill in your values:

.env
cp .env.example .env
# Edit .env:
OPENAI_API_KEY=sk-...
S3_BUCKET=my-rag-bucket
DOCS_PREFIX=docs/
2

Upload your documents & build the FAISS index

Upload your .txt files to the S3 docs/ prefix, then run the indexer. It chunks the text, generates embeddings with OpenAI, and uploads the FAISS index back to S3.

terminal
# Upload your documents
aws s3 cp my-docs/ s3://my-rag-bucket/docs/ --recursive

# Build the FAISS vector index (run once, or when docs change)
python src/indexer.py
💡
The indexer reads each .txt file, splits it into 512-character overlapping chunks, embeds each chunk with text-embedding-3-small, and saves index.faiss + chunks.pkl to S3. This is a one-off step — only re-run when your documents change.
3

Test locally with SAM

Before deploying to AWS, test the Lambda locally. Requires Docker.

terminal
sam build
sam local start-api

# In a second terminal window:
curl -X POST http://127.0.0.1:3000/query \
  -H "Content-Type: application/json" \
  -d '{"question": "What is the refund policy?"}'
4

Deploy to AWS

Deploy the Lambda and API Gateway to your AWS account. The first time use --guided — SAM will ask you a few questions and remember the answers for next time.

terminal
sam deploy --guided

# SAM will prompt for:
#   Stack name        → agentic-rag
#   AWS Region        → us-east-1
#   OpenAIApiKey      → sk-...   (masked)
#   S3Bucket          → my-rag-bucket
#   IndexKey          → (press Enter for default)
#   ChunksKey         → (press Enter for default)
📋
After deployment, SAM prints the ApiUrl. Copy it — you'll need it for the next step. Future deployments just need sam deploy (no --guided).
5

Call the live API

Your RAG agent is now running on AWS. Query it from anywhere.

terminal
API_URL=https://<id>.execute-api.us-east-1.amazonaws.com

curl -X POST "$API_URL/query" \
  -H "Content-Type: application/json" \
  -d '{"question": "Summarise the key return policy points."}'

# Expected response:
# {"answer": "According to the knowledge base, ..."}

# Health check
curl "$API_URL/health"

Bedrock RAG Agent

AWS-Native · Fully Managed · No OpenAI

Everything is managed by AWS. No FAISS, no embedding code, no retrieval logic. You upload documents to S3 and AWS handles chunking, indexing, retrieval, and the agentic loop automatically using Claude 3.5 Sonnet on Bedrock.

🎯
Best for: Production systems where you want minimal maintenance, deep AWS integration, and no dependency on OpenAI. Supports conversation sessions out of the box.
💸
Cost warning: OpenSearch Serverless (used for vector storage) costs approximately $700/month minimum (2 always-on compute units). This project is not suitable for experimentation or low-traffic dev/test — use Agentic RAG or LangGraph RAG for those.

Install local dependencies & deploy infrastructure

Unlike Agentic RAG, there is no FAISS index to build locally. The infrastructure (Bedrock Agent, Knowledge Base, OpenSearch Serverless) is created entirely by SAM.

Bedrock_RAG_Agent/
cd Bedrock_RAG_Agent
pip install -r requirements.txt   # boto3 + python-dotenv

sam build
sam deploy --guided

# SAM will prompt for:
#   Stack name     → bedrock-rag-agent
#   AWS Region     → us-east-1  (must support Bedrock)
#   CollectionName → (press Enter for default)
📋
SAM prints four important outputs after deploy: ApiUrl, DocumentsBucket, KnowledgeBaseId, and DataSourceId. Save these — you'll need them in the next steps.
2

Upload your documents to S3

Upload any supported document format to the S3 bucket that SAM created.

terminal
# Copy the bucket name from SAM outputs
BUCKET=<DocumentsBucket from SAM output>

aws s3 cp my-docs/ s3://$BUCKET/ --recursive
📄
Supported formats: plain text (.txt), PDF, Word (.docx), HTML, Markdown. Bedrock handles all chunking and embedding automatically.
3

Sync the Knowledge Base

After uploading documents, trigger an ingestion job. Bedrock will chunk, embed, and index all documents into OpenSearch Serverless.

terminal
# Fill in KnowledgeBaseId and DataSourceId from SAM outputs
cp .env.example .env
# Edit .env:
#   BEDROCK_KB_ID=<KnowledgeBaseId>
#   BEDROCK_DATA_SOURCE_ID=<DataSourceId>

python scripts/sync_kb.py
# Script polls until ingestion completes and prints progress
🔄
Re-run sync_kb.py every time you add or update documents in S3. It starts an ingestion job and waits for it to finish.
4

Query the API

The Bedrock agent supports multi-turn conversations. Pass the returned session_id back to continue a conversation.

terminal
API_URL=https://<id>.execute-api.us-east-1.amazonaws.com # First question curl -X POST "$API_URL/query" \ -H "Content-Type: application/json" \ -d '{"question": "What is the return policy?"}' # Follow-up in the same session (pass the session_id from above) curl -X POST "$API_URL/query" \ -H "Content-Type: application/json" \ -d '{"question":"Can you elaborate?","session_id":"550e8400-..."}'
💬
The response includes both answer and session_id. Pass the session_id back in follow-up questions to maintain conversation context — a unique feature of the Bedrock project.

LangGraph RAG

LangGraph · FAISS · gpt-4o-mini

The same FAISS + S3 retrieval as Agentic RAG, but the flow is defined as an explicit typed graph using LangGraph. Instead of an open-ended agent loop, the pipeline is a fixed sequence: retrieve → generate. Cheaper, simpler, and easy to extend visually.

🎯
Best for: When you want a structured, visible pipeline that is easy to extend later (add grading, rewriting, routing) without changing the Lambda handler. A natural stepping stone toward CRAG or multi-agent patterns.

Which project should I start with?

Start with Agentic RAG if…
  • You want the most flexible and educational project to learn from
  • You need full control over chunking, retrieval, and search strategy
  • You are cost-conscious (pay only for OpenAI API calls)
  • You want to swap the LLM easily later
Start with LangGraph RAG if…
  • You like Agentic RAG but want a cleaner, more structured codebase
  • You plan to extend the pipeline later (add grading, rewriting, routing)
  • You want to visualise the data flow as a graph
  • You want a cheaper LLM (gpt-4o-mini vs gpt-4o)
Start with Bedrock RAG Agent if…
  • You need a production-grade managed system with minimal code
  • You need multi-turn conversation sessions
  • You cannot use OpenAI (regulatory or preference reasons)
  • You are deeply invested in the AWS ecosystem and IAM security
  • Budget is not a concern (~$700/month floor is acceptable)

Glossary

SAM — Serverless Application Model

An AWS tool that lets you define serverless infrastructure (Lambda, API Gateway, IAM roles) in a template.yaml file and deploy everything with sam deploy.

IAM Role

A set of permissions in AWS. Each Lambda function has a role that defines what AWS services it is allowed to access (e.g., read-only access to a specific S3 bucket).

Function Calling / Tool Use

A feature where you tell an LLM about available functions (tools). The LLM can then choose to call them with arguments, and you run the function and feed the result back.

Chunking

Splitting a large document into smaller overlapping pieces (e.g., 512 characters with 64-character overlap). Each chunk is embedded and stored separately in the vector index.

TypedDict

A Python type hint that gives a dictionary a defined structure. LangGraph uses it for RAGState so the graph state is self-documenting and type-checked.

OpenSearch Serverless

Amazon's managed vector search service. Used by Bedrock Knowledge Base to store and search embeddings. Expensive (~$700/mo minimum) but requires zero maintenance.

HTTP API vs REST API (Gateway)

All three projects use the newer HTTP API Gateway (not the older REST API). It is cheaper, faster, and supports payload format 2.0 which simplifies Lambda handlers.

/tmp in Lambda

Lambda functions have a small writable filesystem at /tmp (up to 10 GB). The FAISS index is downloaded here on cold start and reused on warm starts — avoiding re-downloading on every request.
